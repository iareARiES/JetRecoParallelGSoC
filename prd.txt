Task 2 — Parallel Implementation (parallel-euclid.jl)3.1  Parallelisation StrategyThe AI must implement parallelism using Julia's built-in Threads.@threads macro over the outer loop. This is the correct approach for this GSoC project context (CPU-level shared-memory parallelism).3.1.1  Required Parallel Functionfunction pairwise_distances_parallel(points::AbstractArray{T}) where T    @assert size(points)[2] == 3    n = size(points)[1]    distances = zeros(T, (n, n))    Threads.@threads for i in 1:n        for j in 1:n            dx = points[i, 1] - points[j, 1]            dy = points[i, 2] - points[j, 2]            dz = points[i, 3] - points[j, 3]            distances[i, j] = sqrt(dx^2 + dy^2 + dz^2)        end    end    return distancesend3.1.2  Why This Approach is Safe (No Race Conditions)The AI must include a comment explaining that Threads.@threads on the outer i loop is thread-safe here because each thread writes to a distinct row i of the distances matrix. No two threads ever write to the same memory location simultaneously.3.2  Additional Optimized Variant (Optional but Recommended)After implementing the basic parallel version, also implement an optimized variant that exploits the symmetry of Euclidean distance (dist(i,j) == dist(j,i)):function pairwise_distances_parallel_optimized(points::AbstractArray{T}) where T    @assert size(points)[2] == 3    n = size(points)[1]    distances = zeros(T, (n, n))    Threads.@threads for i in 1:n        for j in i:n   # Only upper triangle            dx = points[i, 1] - points[j, 1]            dy = points[i, 2] - points[j, 2]            dz = points[i, 3] - points[j, 3]            d = sqrt(dx^2 + dy^2 + dz^2)            distances[i, j] = d            distances[j, i] = d   # Symmetry        end    end    return distancesendNote: this optimized variant has a race condition risk on the distances[j, i] write when i != j. The AI must detect this and either: (a) use a per-row atomic approach, (b) only benchmark the non-optimized safe version for the main results, and (c) include the optimized version with a clear comment about its correctness constraints.4. Task 4 — GPU Porting Discussion5.1  What to ProduceA section in README.md (or a separate GPU_DISCUSSION.md) of at least 400 words discussing what would be required to port pairwise_distances to a GPU in Julia. The discussion must cover all the points below.5.2  Required Discussion Points5.2.1  Julia GPU EcosystemCUDA.jl for NVIDIA GPUs (most relevant for CERN hardware).AMDGPU.jl for AMD GPUs.oneAPI.jl for Intel GPUs.Metal.jl for Apple Silicon.Mention that Julia's GPU programming model is high-level via KernelAbstractions.jl for hardware-portable kernels.5.2.2  Memory Considerations (Critical)For n=10,000 points, the output distances matrix is 10000 * 10000 * 4 bytes (Float32) = 400 MB. This must fit in GPU VRAM.Points array is 10000 * 3 * 4 = 120 KB. This is small and fits easily.Data transfer between CPU RAM and GPU VRAM is the primary bottleneck. Must use CuArray or equivalent to keep data on-device across calls.Avoid repeated CPU-GPU transfers. Transfer points once, keep distances on GPU, only copy back if needed.5.2.3  Kernel DesignMap the (i, j) 2D loop to a 2D CUDA thread block grid.Each GPU thread computes one distance: thread (i, j) computes distances[i, j].Total GPU parallelism: n*n = 100 million independent computations, which perfectly saturates GPU warps.Use shared memory tiling to cache point coordinates and reduce global memory accesses.The computation is embarrassingly parallel: no reductions or inter-thread communication needed.5.2.4  Symmetry ExploitationOnly compute upper triangle and mirror. Halves computation but complicates thread assignment.Discuss the tradeoff: simpler full (i,j) grid with no branching vs. triangular grid with branch divergence.For GPU, full grid (no symmetry) with no conditionals often outperforms triangular due to warp efficiency.5.2.5  Numerical PrecisionFloat32 is preferred on GPU for throughput (GPU Float32 throughput is typically 2x Float64).The existing code already uses Float32. Good choice.Mention that sqrt is expensive on GPU; consider using squared distances if only comparison is needed.5.2.6  Example CUDA.jl Sketchusing CUDA function gpu_kernel!(distances, points, n)    i = (blockIdx().x - 1) * blockDim().x + threadIdx().x    j = (blockIdx().y - 1) * blockDim().y + threadIdx().y    if i &lt;= n &amp;&amp; j &lt;= n        dx = points[i, 1] - points[j, 1]        dy = points[i, 2] - points[j, 2]        dz = points[i, 3] - points[j, 3]        distances[i, j] = sqrt(dx^2 + dy^2 + dz^2)    end    returnend function pairwise_distances_gpu(points_cpu)    n = size(points_cpu, 1)    points_gpu = CuArray(points_cpu)    distances_gpu = CUDA.zeros(Float32, n, n)    threads = (16, 16)    blocks = (cld(n, 16), cld(n, 16))    @cuda threads=threads blocks=blocks gpu_kernel!(distances_gpu, points_gpu, n)    return Array(distances_gpu)   # Copy back to CPUend6.
